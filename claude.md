# 项目开发规范

## 代码编写规范

### 基本原则

- 遵循资深开发规范和行业标准
- 采用当前流行的最佳实践
- 符合模块化设计原则

### 代码质量要求

- **简单易懂**：代码逻辑清晰，避免过度设计
- **结构清晰**：模块职责明确，层次分明
- **可维护性**：便于后续扩展和维护

### 注释规范

- 为重要的模块、类、复杂方法添加清晰的注释
- 减少非必要注释，代码应该是自解释的
- 注释应说明"为什么"而非"是什么"

### 模块化实践

- 合理划分功能模块
- 保持模块间低耦合、高内聚
- 复用性强的逻辑抽取为独立工具函数

## Git 提交规范

### 提交规范

- 遵循 Conventional Commits 规范
- 格式：`<type>: <subject>`
  - type: feat, fix, docs, style, refactor, test, chore 等
  - subject: 简洁描述，不超过 50 字符

### 提交行为

- **禁止自动提交**：除非用户明确要求，否则不要执行 git commit
- **纯人工提交信息**：提交日志中不包含任何 AI 相关内容（如 "Generated with Claude Code" 等）
- **提交前确认**：提交前应展示改动内容，等待用户确认
